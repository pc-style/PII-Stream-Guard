The leak

How it happens

01

The stream

You go live. Your email sits in the corner of the frame for two hours. By the end of the stream it belongs to everyone watching.

02

The client demo

Screen-sharing with a client. Your personal Gmail is right there in the tab bar. Very professional.

03

The tutorial

You record a walkthrough. A customer's phone number is visible for fourteen minutes. You find out after it is published.

04

The handoff

You share a log file with an internal string in it. It is now in three Slack channels you do not own.

The fix

Catch it before anyone else does

core principle

PII Stream Guard watches your screen, finds what looks like personal data, and boxes or blacks it out on a delayed preview before it ever reaches the stream.

Capture and detection are decoupled. The screen is captured continuously, OCR runs on a throttled background queue, and the preview only reads the latest matched boxes.

If the remote processing path ever fails, the client fails closed and shows a black frame. No silent leak path.

Pipeline

How it works

01 / capture
ScreenCaptureKit

Main display to CVPixelBuffer, never blocks OCR.

02 / throttle
Background queue

Downscaled frame, 5 to 10 fps, off the hot path.

03 / detect
Vision OCR

Email regex + needles, per-substring boxes.

04 / store
PIIBox matches

Thread-safe snapshot of the latest boxes.

05 / render
Protected preview

Boxes or blackout on the delayed frame.

Modes

Pick your caution

Lockdown

For moments you cannot afford a leak. Accurate OCR, a longer delay buffer, and blackouts drawn while armed. Use this for client demos, recordings, and anything you cannot re-shoot.

accurate OCR higher delay blackout while armed enhance low contrast

Standard

The balanced default for normal screen sharing. Fast OCR, a modest delay, and bounding boxes by default with blackout one toggle away.

fast OCR moderate delay boxes by default

Low-latency

When responsiveness matters more than perfect safety. Lower delay and a faster detection cadence. Good for live walkthroughs where you want the preview to track your actions closely.

lowest delay faster cadence boxes by default
Topology

Two ways to run it

Local

One Mac, all of it

The same machine captures the screen, runs OCR, and renders the protected preview. Zero network, zero trust surface. The simplest path.

Remote

Client and server

A client Mac captures frames and sends them to a processing server over a token-authenticated WebSocket. The server returns protected frames. If the remote path fails, the client fails closed with a black frame.

Install

Up and running in a minute

Requirements

macOS only. Built with SwiftPM, no Xcode project. The interactive installer builds the release binary and offers to start any mode.

  • macOS 14 or newer
  • SwiftPM from Xcode or Command Line Tools
  • Screen Recording permission for your terminal
one-line installer./scripts/setup.sh
build from sourceswift build -c release
.build/release/pii-stream watch --mode standard
remote: server + clientpii-stream serve --host 0.0.0.0 --port 8765 \
  --token "replace-with-a-long-random-token"

pii-stream watch --remote SERVER:8765 \
  --token "replace-with-a-long-random-token" --mode standard
scan a static imagepii-stream detect-image --image ./screenshot.png --json

If anything in the pipeline breaks, you do not get a partial frame with PII in it. You get black. The safe default is no picture.
